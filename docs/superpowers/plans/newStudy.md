## 二、后续学习顺序调整原则

新的顺序不再直接从“阶段二主线完成”跳到“阶段三”，而是改成：

1. 先补完阶段二剩余扩展项
2. 再进入阶段三：数据上报与 SDK 内部事件流
3. 再进入阶段四：错误、白屏、录屏回放
4. 再进入阶段五：性能与 `web-vitals`
5. 最后进入阶段六：框架无关 SDK 与工程治理

这样调整的原因：

- 当前项目已经具备基础埋点链路，但业务埋点的高级场景还没补全
- 阶段三、四、五、六更偏“体系升级”，应该建立在阶段二完整收口之上
- 这样学下来更符合真实项目的演进顺序

## 三、阶段二补完计划

### 2.1 无埋点 / 全埋点

目标：

- 理解为什么“全采”看似方便，实际非常容易带来噪音、性能负担和治理成本
- 明确手动埋点、声明式埋点、可视化埋点、无埋点之间的职责边界

计划内容：

- 什么是无埋点 / 全埋点
- 它适合采什么，不适合采什么
- 为什么无埋点不能替代业务语义埋点
- 在当前项目里模拟“自动点击采集”和“可治理业务埋点”的区别

### 2.2 提交、下载类事件

目标：

- 让阶段二不只停留在“曝光 + 点击 + 加购”
- 补足常见业务事件类型

计划内容：

- 表单提交事件
- 文件下载事件
- 提交成功 / 失败区分
- 下载目标、来源、位置等业务字段设计

### 2.3 UML 时序图与事件流文档

目标：

- 把“用户操作 -> SDK 采集 -> queue -> transport -> server”的链路正式画出来

计划内容：

- 基于当前项目代码产出时序图
- 把曝光、点击、加购分别映射到事件流里

### 2.4 RBAC 鉴权失败埋点

目标：

- 学习业务异常不只有 JS Error，也包括“用户流程被业务权限拦截”

计划内容：

- 模拟一个受权限控制的操作
- 记录角色、资源、动作、失败原因
- 区分“前端按钮不可见”和“点击后被拒绝”

### 2.5 多步表单转化链路

目标：

- 补足“登录 -> 填写 -> 确认 -> 提交成功/失败”这类真实流程型埋点

计划内容：

- 设计一个最小多步流程页
- 记录每一步进入、退出、提交、失败
- 计算阶段转化率和流失率

### 2.6 防抖、去重、采样、重复上报拦截

目标：

- 学习为什么埋点系统不只是“采到就发”

计划内容：

- 高频点击防抖
- 曝光去重
- 错误采样
- 重复事件拦截条件设计

### 2.7 事件字典、埋点文档、版本管理

目标：

- 补齐阶段二的治理意识

计划内容：

- 事件字典结构设计
- 公共字段和业务字段文档化
- 事件命名规范
- 版本变更记录方式

### 2.8 完整登录到下单埋点体系

目标：

- 让阶段二最终产出一个真正像样的业务埋点方案

计划内容：

- 登录页
- 商品浏览页
- 商品详情页
- 加购
- 下单
- 结果页
- 转化率计算

阶段二补完后，才正式进入阶段三。
