# 监控与埋点 Demo 设计文档

## 一、项目概述

这是一个面向一天内学习与落地演示的前端监控系统教学项目，技术栈为 `Vue 3 + TypeScript`。项目目标不是复刻完整的大厂监控平台，而是在可控范围内做出一个“小而完整”的 Demo，让你真正理解前端监控与埋点的核心概念、实现方式和工程设计思路。

整个项目包含两大部分：

1. 一个用于制造真实监控场景的 Vue 业务演示应用
2. 一个用于采集、规范化、入队和上报事件的迷你监控 SDK

这个项目的学习目标是：

- 在一天内掌握前端监控与埋点的核心知识框架
- 做出一个可运行、可演示、可讲解的 Demo
- 在有限范围内建立接近大厂项目的工程拆分和设计思维

## 二、学习范围

本次 Demo 聚焦最值得优先学习的前端监控能力：

- JavaScript 运行时错误监控
- Promise 异常监控
- 资源加载失败监控
- 页面访问与路由切换埋点
- 点击行为埋点
- 页面停留时长统计
- 基础性能时序采集
- 统一事件模型设计
- 公共上下文字段补充
- 队列、去重、节流、批量上报
- 本地事件可视化，便于调试与教学

以下内容明确不纳入本次一天版本范围：

- 完整的后端存储与分析平台
- 数据库接入
- 图表大盘系统
- 用户会话回放
- source map 反解
- 全量 Web Vitals 指标与深入性能优化
- 分布式链路追踪与报警平台

这样做的原因不是这些内容不重要，而是它们会显著分散一天内的学习收益。当前阶段我们优先保证“前端侧监控核心链路”被真正学会并做出来。

## 三、最终产物形态

最终 Demo 应该呈现为一个“小而完整”的系统，而不是零散功能拼接。

### 1. 业务演示应用

Vue 应用负责制造真实监控场景，并为学习过程提供直观反馈。计划包含以下页面或能力区域：

- 首页：介绍本项目支持的监控能力
- 错误演示页：触发运行时错误、Promise 异常、资源加载失败
- 埋点演示页：触发点击、路由访问、页面停留等行为事件
- 性能演示页：展示浏览器性能时序信息
- 事件面板页：展示已经被上报或接收的监控事件

### 2. 迷你监控 SDK

SDK 是本项目的核心。它不是写在页面里的零散脚本，而是一层可复用的监控能力封装。它会对外暴露少量清晰的 API，并在内部组织采集、格式化、缓存、去重和上报逻辑。

预期对外 API 示意：

- `initMonitor(config)`
- `track(eventName, payload)`

预期内部职责包括：

- 注册浏览器监听器
- 构建统一事件结构
- 补齐公共上下文，例如页面地址、时间戳、应用标识、环境、用户或会话标识
- 将事件加入队列并批量发送
- 避免明显的重复上报
- 将数据发送到本地 mock 服务

## 四、整体架构设计

### 1. 前端结构

前端使用 `Vue 3 + TypeScript` 实现，目录结构按职责拆分，而不是把所有内容堆在一起。

推荐结构如下：

```text
src/
  app/
  components/
  pages/
  router/
  sdk/
    collectors/
    core/
    transport/
    types/
    utils/
  styles/
```

各目录职责如下：

- `app/`：应用装配层，负责启动、初始化、监控注册等逻辑
- `components/`：可复用的展示组件与交互组件
- `pages/`：围绕教学场景设计的页面
- `router/`：路由注册，以及后续基于路由的埋点逻辑
- `sdk/core/`：监控入口、初始化、生命周期管理
- `sdk/collectors/`：按信号类型拆分的采集器
- `sdk/transport/`：事件队列、上报与发送策略
- `sdk/types/`：事件类型与配置类型定义
- `sdk/utils/`：通用工具函数

### 2. `App.vue` 与 `src/app` 的关系

项目仍然会保留 `App.vue` 作为 Vue 根组件，它负责最外层的页面壳子与视图承载。

与此同时，`src/app` 目录用于承载“应用装配层”逻辑，例如：

- 创建 Vue 应用
- 注册路由
- 注册全局依赖
- 初始化监控 SDK
- 启动时的工程级装配逻辑

这样拆分的目的，是让“页面视图”和“应用启动逻辑”职责分离，更贴近中大型项目实践，也更便于教学。

### 3. Mock 后端

项目会使用轻量的 `Node.js + Express` 作为本地 mock 接收端，用来模拟监控上报链路。

它只承担非常有限的职责：

- 提供 `POST /api/report` 接收上报事件
- 提供 `GET /api/events` 返回已接收事件
- 使用内存数组存储数据，不接数据库
- 提供基础的 JSON 解析和本地开发支持

它的存在是为了帮助我们打通“前端采集 -> HTTP 上报 -> 服务端接收 -> 前端展示”这条完整链路，而不是把学习重点转移到后端开发。

## 五、事件模型设计

SDK 会把不同来源的监控信号统一整理成一致的数据结构。具体字段在实现中可以有少量微调，但整体方向固定。

核心字段建议包括：

- `id`：事件唯一标识
- `type`：事件大类，例如错误、性能、行为
- `subType`：事件子类型，例如 `js_error`、`route_change`
- `name`：事件名称或标签
- `timestamp`：事件产生时间
- `url`：当前页面地址
- `appId`：应用标识
- `sessionId`：当前会话标识
- `payload`：当前事件专属的明细内容

统一事件模型的意义在于：

- 不同采集器可以共用同一套上报管道
- 前端展示时更容易统一渲染
- 后续扩展新事件类型时结构更稳定
- 更容易体现工程化思维，而不是零散拼接

## 六、上报策略设计

上报层不仅仅是“调用一次 fetch”，而是要体现一些基础工程考虑。为了兼顾教学价值与当天可完成性，本项目会实现一套简化但合理的传输策略。

计划包含：

- 内存队列
- 基础节流
- 批量发送
- 对短时间内相同错误做简单去重
- 基于 `fetch` 的上报方式
- 为后续扩展 `sendBeacon` 预留空间
- 在本地调试场景下尽量保证演示稳定

这样你能学到：监控系统真正有价值的，不只是“监听到了什么”，还包括“这些数据怎样可靠地组织和发出去”。

## 七、教学与实现顺序

实现顺序会严格围绕“一天内学会并做出 Demo”来安排。

### 阶段 1：项目启动与骨架搭建

完成 `Vue 3 + TypeScript` 前端项目和轻量 `Express` mock 服务搭建。

这个阶段的目标是：

- 项目能跑起来
- 基础目录结构清晰
- 前后端本地联调环境准备好

### 阶段 2：先打通最小可用链路

优先完成：

- 监控初始化
- 手动 `track()`
- 上报接口联通
- 事件展示页

这一步会先把“从采集到展示”的最短闭环打通，避免后面采集能力越写越多，但整体链路还没成型。

### 阶段 3：接入自动采集器

在最小链路跑通后，逐步补齐以下自动采集能力：

- JavaScript 错误采集
- Promise 异常采集
- 资源加载失败采集
- 路由访问采集
- 点击行为采集
- 页面停留时长采集

这样做的好处是：你会很自然地理解“每一种监控能力，本质上都是一个 collector”。

### 阶段 4：补充性能采集

把浏览器性能数据作为另一类事件接入统一模型。

这样你会同时学到：

- 为什么浏览器能提供这些性能信息
- 这些指标和业务有什么关系
- 为什么性能监控也应该走统一事件管道

### 阶段 5：做成可展示 Demo

最后补齐演示体验，包括：

- 可手动触发的错误和埋点按钮
- 可视化事件展示面板
- 更适合教学与演示的页面结构

这一步的目标是把项目从“学习代码”提升为“可以拿出来展示的作品”。

## 八、错误处理原则

即使这是教学项目，也应体现良好的前端工程习惯：

- 非法或缺失事件字段不能导致页面崩溃
- 某个采集器自身出错时要安全失败
- 上报失败不能影响主业务页面使用
- 事件展示页需要具备加载态、空态和基础异常处理

## 九、验证方式

考虑到这是一天版本，验证重点放在“关键链路真实跑通”，而不是追求非常完整的自动化测试覆盖。

必须验证的结果包括：

- 前端应用能成功启动
- Mock 服务能成功启动
- 手动埋点能成功到达 mock 服务
- 错误事件可以被触发并看到
- 性能事件可以被采集并展示
- 事件面板能正确展示接收到的数据

如果时间允许，可以补充少量针对工具函数或事件格式化逻辑的轻量测试，但这不是第一优先级。

## 十、成功标准

如果在今天结束时，你可以做到以下几点，就说明这次学习是成功的：

- 能解释监控 SDK 在前端中的作用
- 能说明为什么监控逻辑不能直接散落在业务页面里
- 能演示一个可运行的 Vue 监控 Demo
- 能讲清楚事件模型和上报链路
- 能展示本地 mock 接收端如何工作
- 能用较清晰的工程语言解释当前方案的取舍

## 十一、实现约束

- 保持项目紧凑、可讲解、可完成
- 优先保证清晰度，不做过度抽象
- 后端只做最小接收能力，不扩散主线
- 设计尽量体现中大型团队的思考方式，但不过度工程化
- 为后续继续扩展保留空间
